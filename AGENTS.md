# AGENTS.md — guide for coding agents working on ramp-up-pack-service

Reactive microservice for **WMA AAT STAAT retention-pack email delivery**: when ACE relationships
are reassigned after FA attrition, the service emails each receiving FA (or the Field Leader for
pended relationships) one HTML retention pack per ACE relationship via the firm **StaatEmail** API —
live or scheduled, with 35 MB splitting, failure notification and a 7-day retry window.
`requirement.md` in the repo root is the business spec; `README.md` covers running it.

## Stack (pinned — do not "upgrade" casually)

Java **21** · Spring Boot **4.1.0** (Framework 7) · **WebFlux** (Netty) · Spring Data **R2DBC** +
`r2dbc-postgresql` · PostgreSQL 16 (Azure, Entra passwordless) · springdoc **3.0.3** ·
azure-identity **1.18.3** · Jackson **3** (`tools.jackson`) · Zonky embedded-postgres (tests).
**No Lombok. No Spring Cloud. No Flyway/Liquibase.**

## Commands

```bash
mvn compile                 # compile
mvn test                    # full suite — embedded PostgreSQL, no Docker, no Azure needed
mvn spotless:apply          # format (palantir-java-format; spotless:check enforced at verify)
mvn spring-boot:run         # run (see README/.env for env vars; profile via SPRING_PROFILES_ACTIVE)
# interactive dev DB on port 5433 with schema+seed applied:
mvn -q org.codehaus.mojo:exec-maven-plugin:3.5.0:java -Dexec.classpathScope=test \
    -Dexec.mainClass=com.ubs.wma.aat.rampuppack.EmbeddedPostgresRunner
```

A change is done only when `mvn test` is green. There is no Docker on this machine — Zonky runs
real PG binaries as a subprocess; never reintroduce Testcontainers.

## Hard policies (never violate)

1. **The app NEVER executes SQL/DDL outside tests.** `spring.sql.init.mode=never` stays in
   `application.yml`. Schema lives ONLY in `src/test/resources/db/schema.sql` (also the DBA handoff
   document) and seed data in `db/seed.sql`. Do not add Flyway/Liquibase, do not enable SQL init in
   non-test profiles, do not put DDL/DML strings in Java — SQL belongs in those files.
2. **Tests never touch the Azure database.** Every integration test extends
   `AbstractIntegrationTest`, which unconditionally points `spring.r2dbc.url` at the embedded
   localhost PG. Never weaken that override, never put an Azure host in test config.
3. **No credentials in the repo.** Config uses `${ENV_VAR:}` placeholders; `.env` is git-ignored.
   Never commit, print or hard-code secrets; redact values when displaying `.env`.

## Database conventions

- Two schemas, mapped explicitly with `@Table(schema = "...", name = "...")` — never rely on
  `search_path`:
  - `aat_app` — owned by this service: `email_template`, `email_batch`, `email_log` (full DML)
  - `datamesh` — externally populated data products: `staat_insight_document` (**SELECT-only**;
    this service must never write to it — seed rows only via `db/seed.sql` in tests)
- Naming: snake_case columns, `BIGINT GENERATED BY DEFAULT AS IDENTITY` ids, `TIMESTAMPTZ` audit
  columns, enums stored as `VARCHAR`.
- Standard audit columns on owned tables: `created_by`/`updated_by`/`created_at`/`updated_at`,
  populated by Spring Data auditing. The auditor comes from the `X-User-Id` request header
  (`AuditingConfig`: WebFilter → Reactor context → `ReactiveAuditorAware`), defaulting to `system`.
  New entities must carry the four `@CreatedBy/@LastModifiedBy/@CreatedDate/@LastModifiedDate`
  fields or inserts fail on NOT NULL.
- Type mapping: `TEXT[]` ↔ `List<String>` is native to PostgresDialect; `JSONB` ↔
  `Map<String,String>` and each enum need explicit converters registered in `R2dbcConfig`
  (Jackson 3: `tools.jackson.databind.json.JsonMapper`, NOT `com.fasterxml`). Adding an enum
  column ⇒ add a reading + writing converter pair.
- Concurrency-safe work claiming uses single-statement
  `UPDATE … WHERE id IN (SELECT … FOR UPDATE SKIP LOCKED) RETURNING *` repository `@Query`s —
  schema-qualify table names in every custom query. Keep this pattern for anything a scheduler
  picks up; do not replace it with read-then-update.
- Changing a table = update `db/schema.sql` + the record entity + (if seeded) `db/seed.sql`,
  keeping both scripts **idempotent** (`IF NOT EXISTS`, `ON CONFLICT DO NOTHING`) — multiple test
  contexts share one embedded PG and re-run them.

## Code conventions

- Layered packages under `com.ubs.wma.aat.rampuppack`: `controller/` → `service/` → `repository/`,
  plus `domain/` (record entities + enums), `dto/` (request/response records), `mapper/` (static
  final classes), `exception/`, `config/` (+`properties/` for `@ConfigurationProperties` records),
  `client/staatemail/` (declarative `@HttpExchange` + DTOs). Follow the existing examples.
- Entities and DTOs are **immutable Java records**. Entities mutate via with-er/factory methods
  (`asSent(...)`, `newTemplate(...)`); a `null` id marks a new row.
- Validation: jakarta annotations on DTO records; cross-field rules via a public
  `@AssertTrue boolean isXxx()` method (mark it `@Schema(hidden = true)`).
- Errors: throw domain exceptions and map them in `GlobalExceptionHandler` to RFC 9457
  `ProblemDetail` (`application/problem+json`). Don't return ad-hoc error bodies.
- **Readability over clever chaining** (explicit owner preference): separate I/O from logic —
  e.g. `EmailSendService` is `loadContext` (I/O) → `planParts` (pure Java, no Reactor) →
  `deliverPart` (short reactive steps). Prefer small named methods and records over nested
  lambdas; keep reactive pipelines shallow.
- Reactive rules: never block on event-loop threads. `.block(timeout)` is acceptable only inside
  `@Scheduled` methods (scheduler thread). Sequential delivery uses `concatMap`, not `flatMap`.
- Document the *why* in Javadoc on classes and non-obvious methods, matching the existing density.

## Spring Boot 4 gotchas (cost real debugging time — respect them)

- `WebClient.Builder` is **not** an auto-configured bean — use `WebClient.builder()` directly.
- Jackson 3 packages are `tools.jackson.*`; checked `JsonProcessingException` is gone.
- Test slices are per-module artifacts/packages: `@DataR2dbcTest` from
  `org.springframework.boot.data.r2dbc.test.autoconfigure`, `@AutoConfigureWebTestClient` from
  `org.springframework.boot.webtestclient.autoconfigure`. Use `@MockitoBean` (not `@MockBean`).
- springdoc 3.x line only (2.x breaks on Boot 4). Spring Cloud stays out; `bootstrap.yaml` is a
  plain imported config file.
- Test slices do not load custom `@Configuration` — repository tests must import
  `{R2dbcConfig, AuditingConfig}` (already done in `RepositoryTestSupport`; extend it).

## Azure / StaatEmail

- Shared identity: `AzureCredentialConfig` resolves SPN (all three `AZURE_*`) → UAMI (client-id
  only) → `DefaultAzureCredential` (local `az login`). DB auth is token-as-password R2DBC
  passwordless (`R2dbcPasswordlessConfig`) — don't replace it; there is no official R2DBC
  passwordless starter.
- StaatEmail has its **own dedicated SPN** (`app.staatemail.tenant-id/client-id/client-secret`,
  client-credentials flow via `ClientSecretCredential`/MSAL), falling back to the shared
  credential when unset. Tokens are attached by `EntraBearerExchangeFilter`; client code never
  handles tokens.
- StaatEmail send protocol is two-step: `POST /uploadAttachment` per attachment → `POST /sendEmail`
  referencing the returned ids. `smeId` is a fresh GUID per attempt and is stored in
  `email_log.sme_id` for tracking.

## Testing conventions

- Controller tests: `@WebFluxTest(XxxController.class)` (from
  `org.springframework.boot.webflux.test.autoconfigure`) + `@MockitoBean` service — test the
  web contract only: routing, status codes, `@Valid` failures (assert 400 + the service had no
  interactions), enum/param binding, ProblemDetail mapping (the `@RestControllerAdvice` is part
  of the slice). Don't re-test business logic there; that belongs to `EmailDeliveryFlowTest`.
  Request bodies are raw JSON text blocks (catches field renames); validation-failure cases are
  one `@ParameterizedTest(name = "{0}")` + `@MethodSource` matrix of (description, body,
  expected-detail-substring) per endpoint — add a new row, not a new test method.
- Entity fixtures for slice/unit tests come from `TestData` (root test package): fully populated
  records with fixed timestamps. Don't use it for tests that persist rows — those keep creating
  their own data with unique marker values.
- Repository tests: extend `repository/RepositoryTestSupport` (`@DataR2dbcTest`, imports, applies
  `db/schema.sql` + `db/seed.sql` per test). R2DBC slice tests do **not** roll back — use unique
  marker values (template codes, faIds) per test and write containment assertions, not exact
  counts; the embedded PG is shared by all contexts in the JVM.
- End-to-end: `@SpringBootTest` + `WebTestClient` + `@MockitoBean StaatEmailClient` (see
  `EmailDeliveryFlowTest`). Only the outbound client is mocked — controller→service→repo→PG is real.
- The scheduler bean is disabled in the test profile (`app.scheduler.enabled=false`); tests build
  `EmailDeliveryScheduler` by hand and call its public pass methods.
- Mockito: to re-stub a `willAnswer` mock mid-test use `willAnswer(...).given(mock).method(any())`;
  `given(mock.method(any()))` re-invokes the previous answer with null args and NPEs.
- Pure logic (merging, splitting) gets plain unit tests — no Spring context.

## Do / Don't quick list

**Do**
- Run `mvn test` before declaring anything done; add tests with the feature.
- Mirror every schema change into `db/schema.sql` (+ seed) and the record entity together.
- Keep new endpoints under `/api/v1/...` with springdoc `@Tag`/`@Operation` and `@Valid` DTOs.
- Make config knobs `@ConfigurationProperties` records with `${ENV_VAR:default}` yml entries.
- Update `README.md` when APIs, env vars or layouts change.

**Don't**
- Don't add Lombok, Spring Cloud, Flyway/Liquibase, Testcontainers or a template engine
  (merge fields are deliberate plain `{name}` string replacement in `TemplateMerger`).
- Don't write to `datamesh.*` from application code, ever.
- Don't run DDL/DML from the app, enable SQL init outside the test profile, or point any config
  default at an Azure host.
- Don't call `.block()` outside `@Scheduled` passes or tests; don't deep-nest reactive chains.
- Don't log secrets, tokens or full email bodies; failure reasons are truncated to 2000 chars.
- Don't "fix" the schedulers' startup `UPDATE … RETURNING` queries as a bug — the first
  `fixedDelay` tick at boot is expected claim behaviour (add `initialDelayString` only if asked).
